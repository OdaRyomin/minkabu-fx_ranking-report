# Build Data Driven Reports

## 目的

構造化データと周辺メモから、読者が「現状」「理由」「差分」「次の打ち手」を理解できるレポートを作る。ジャンル固有の前提に閉じず、まずデータの意味、読者、比較軸、開示範囲を整理してから成果物へ落とし込む。

## 最初に確認すること

ユーザーがすでに指定している情報は聞き直さない。不足していて作業品質に影響する場合だけ、最大3問で確認する。

- 対象: どの会社、商品、地域、期間、セグメントのレポートか
- 読者: 社内向け、顧客向け、営業資料、経営報告、公開記事のどれか
- 成果物: Markdown、HTML、PowerPoint、Excel、PDF想定など
- データ: 参照すべきファイル、フォルダ、月次データの保存場所
- 開示制約: 非公開にすべき配点、係数、内部ロジック、個人情報、契約情報

情報が足りなくても、合理的な仮定を置いて進める。仮定は成果物か最終報告に短く明記する。

## 基本ワークフロー

### Step 1. データ棚卸し

- 対象フォルダのファイル一覧を確認する
- 各ファイルの役割を推定する
- 主データ、補助データ、コメント/メモ、更新履歴、マスタ情報を分ける
- CSV/Excelはヘッダー、サンプル行、欠損、単位、期間を確認する
- 同じ名前の月次ファイルがある場合は、時系列スナップショットとして扱えるか確認する

出力前に、どのファイルを何に使ったかを説明できる状態にする。

### Step 2. レポートの意思決定軸を決める

レポートは数字を並べるだけで終わらせない。最初に次を固定する。

- 主指標: 最初に見せる結論の数字
- 比較軸: 前月比、前期比、上位平均、競合、目標、カテゴリ平均との差など
- 内訳: カテゴリ別、商品別、チャネル別、地域別、月別など
- 解釈: なぜその結果なのか
- 次の打ち手: 改善余地、注視点、提案、継続観察

ランキングや評価データでは、総合順位だけでなく「どのカテゴリが順位を押し上げ/押し下げたか」を必ず見る。

### Step 3. 指標を設計する

定量パートはダッシュボードとして設計する。指標は次の層で整理する。

- Core Metrics: 総合順位、総合点、売上、CV、成長率など最重要指標
- Diagnostic Metrics: カテゴリ別点数、項目別差分、構成比、弱点/強み
- Trend Metrics: 月次推移、半期推移、前年同月比
- Benchmark Metrics: 上位平均、競合、目標、業界平均
- Evidence Metrics: 更新履歴、施策件数、最終更新日、コメント数

よい指標の条件:

- 読者が理解できる
- 比較できる
- 行動や判断につながる
- 単位と期間が明確

### Step 4. 定性情報を接続する

コメント、更新履歴、営業メモ、編集部所見、ユーザーの補足は、数値の解釈に使う。

- 数字で見える事実
- その事実を説明する更新や施策
- 読者にとっての意味
- 次に見るべき観点

この4層を接続して書く。定性コメントだけで順位や成果を説明しない。

### Step 5. レポート構成を作る

標準構成:

1. 表紙 / 対象範囲 / 更新日
2. エグゼクティブサマリー
3. 主要KPI
4. 比較 / ベンチマーク
5. 推移
6. 内訳 / カテゴリ診断
7. 更新履歴 / 根拠
8. 所見 / 改善余地
9. 評価軸 / 定義 / 注記

用途別の補正:

- 顧客向け: 開示範囲を絞り、説明責任と納得感を優先する
- 社内向け: 内部メモ、判断材料、未確定事項も残す
- 営業向け: 商談で話す順番に合わせ、強みと次アクションを明確にする
- 公開記事向け: 読者メリットと根拠の透明性を優先する

### Step 6. デザインと可視化方針を固める

HTML、スライド、ダッシュボード系レポートでは、実装前に `DESIGN.md` を確認する。存在しない場合は、成果物専用の `DESIGN.md` を作る。

デザイン方針:

- 最初に全体像、その次に比較、そのあとに詳細を置く
- KPIは単独で大きく置かず、差分や参照値を近くに置く
- 色は意味で使い、カテゴリごとに安易に虹色を振らない
- 更新日、定義、出典、注記を必ず設計に含める

推奨グラフ:

- 推移: 折れ線
- カテゴリ比較: 横棒、dumbbell chart、スモールマルチプル
- ランキング: 横棒ランキング
- 更新履歴: タイムラインまたは表
- 正確値確認: テーブル

避ける表現:

- 3Dグラフ
- 過度なゲージ
- 読み取り精度が落ちるレーダーチャートの多用
- 色だけで良し悪しを伝える表現

## 関連スキルの併用

必要に応じて、以下を併用する。

- `ranking-data-deep-analyst`: ランキング/評価データから、順位理由や競合差分を深掘りする
- `information-design-structure-builder`: レポート、LP、ダッシュボードの情報構造を先に固める
- `information-design-visual-composer`: デザイン仕様、配色、グリッド、コンポーネント、グラフ表現を詰める
- `metrics-dashboard`: 指標体系、データソース、可視化、レビュー観点を設計する
- `xlsx`: Excelデータの作成、編集、集計、可視化が必要なとき
- `pptx`: PowerPointレポートや営業資料にする必要があるとき

## 開示制御

レポート作成時は、必ず「見せる情報」と「見せない情報」を分ける。

見せやすい情報:

- 評価軸
- 結果スコア
- 順位
- 比較差分
- 更新履歴
- 編集部/分析者の所見

慎重に扱う情報:

- 詳細配点
- 調整係数
- 内部補正ロジック
- 報酬単価や契約条件
- 個人情報
- 社外秘メモ

顧客向けレポートでは「結果と評価軸は見せるが、細かな内部算出ロジックは出しすぎない」を基本にする。

## 品質チェック

完了前に確認する。

- データソースと対象期間が明記されている
- 主指標、比較軸、内訳が混ざっていない
- 数字に単位、期間、定義がある
- 数値とコメントが接続されている
- 重要な欠損や前提が注記されている
- 社外秘情報を出しすぎていない
- 読者が次に何を見るべきか分かる
- HTMLやスライドの場合、モバイル/印刷/スクリーンショット共有で破綻しない

## 出力時の保存ルール

- ユーザーが保存先を指定したら従う
- 指定がない場合は、入力データ近くに `レポート/` または `reports/` を作ることを提案し、既存フォルダがあればそこに保存する
- ファイル名には対象、期間、用途を入れる
- 例: `ヒロセ通商_LIONFX_半期評価レポート_2026H1.md`
- 例: `securities_companyA_quarterly_report_2026Q1.html`
