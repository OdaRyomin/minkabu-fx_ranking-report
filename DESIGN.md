# DESIGN.md

## 0. このドキュメントの役割

この `DESIGN.md` は、同ディレクトリの `index.html` を正本とする実装連動型のデザイン仕様である。

- 対象ファイル: `index.html`
- 対象ページ: みんかぶFX 半期評価レポート
- 想定用途:
  - 同一フォーマットで別FX会社版を量産する
  - HTMLの見た目を崩さずに文言や数値を差し替える
  - デザイン修正時に、何を守り何を変えてよいかを明確にする

この文書では、抽象的なデザイン思想よりも、現在の完成版ページの見た目と構造を再現・保守するためのルールを優先する。

---

## 1. デザイン要約

このページの方向性は `Editorial Financial Report Surface` とする。

- ヒーローだけを濃紺で強く見せ、レポート商品としての格を作る
- 本文は白い紙面のように整理し、営業資料より「説明責任のある分析レポート」に寄せる
- 数字は強く、本文は静かに、比較は読みやすく見せる
- 色は意味で使い、面白さのために増やさない
- KPI、比較、詳細、所見、注記の順で、1ページ完結の縦長レポートにまとめる

避けること:

- トレードアプリ風のネオン表現
- 何でもカード化して強弱がなくなる構成
- 過剰なアニメーション
- カテゴリごとに色を増やすこと

## 2. 体験目標

### 主な読者

- FX会社のマーケティング担当
- 事業責任者
- サービス企画 / プロダクト担当
- 広報 / アライアンス担当

### 読後に持ってほしい理解

- 総合順位と総合スコア
- 強みと改善余地
- 上位社との距離
- 何がその順位を作っているのか
- 半年間でどこが評価に効いたのか

### このページの感情設計

- 第一印象: 信頼感、格、分析資料感
- 本文読了時: 納得感、説明可能性、改善論点の明確さ

## 3. 情報設計

ページは以下の順で読む設計とする。

1. ヒーロー
2. 主要KPIカード
3. エグゼクティブサマリー
4. 総合評価サマリー
5. 競争ポジション
6. カテゴリ診断
7. 主要評価ポイント
8. 更新事項
9. 編集部所見
10. 留意事項とデータ出典

情報順の意図:

- まず「いま何位で、何が強いか」を即答する
- 次に「なぜその順位なのか」をテーブルと比較で補足する
- 最後に編集部所見と注意書きで解釈を閉じる

## 4. ビジュアルシステム

### 4-1. カラー

現在の実装で使う主な色は以下。

| Token | Value | 用途 |
| --- | --- | --- |
| `--report-navy-950` | `#0e1a2b` | ヒーロー背景の最深部 |
| `--report-navy-900` | `#102749` | ヒーロー中間色 |
| `--report-navy-800` | `#11325d` | ヒーロー右側の明るい層 |
| `--signal-blue-700` | `#155eef` | 主役の青。順位、KPI数値、セクション番号 |
| `--signal-blue-600` | `#1d4ed8` | グラデーション終点、補助青 |
| `--signal-blue-050` | `#eef4ff` | 淡い青背景 |
| `--teal-600` | `#0f8b8d` | 比較系列、補助比較色 |
| `--teal-500` | `#169aac` | 平均値バー、比較値 |
| `--minkabu-red-600` | `#e62811` | サンプルバッジなど局所アクセント |
| `--green-600` | `#2f855a` | プラス差分、強み |
| `--amber-600` | `#b7791f` | 継続観察、今後反映しやすい観点 |
| `--red-600` | `#c24141` | マイナス差分 |
| `--line-200` | `#e2e8f0` | 区切り線 |
| `--line-300` | `#cbd5e1` | パネル枠、テーブル線 |
| `--ink-900` | `#1e293b` | 本文主色 |
| `--ink-700` | `#475569` | 補助本文 |
| `--ink-500` | `#64748b` | 注記寄りのトーン |

色の運用ルール:

- 主役は `Navy + Blue` で成立させる
- `Teal` は比較相手専用に近い扱いとする
- `Red / Green / Amber` は意味色としてのみ使う
- `Minkabu Red` はヒーロー右上など局所アクセントに限定する

### 4-2. タイポグラフィ

フォントは以下の2系統で運用する。

- 日本語本文: `Zen Kaku Gothic New`, `Hiragino Sans`, `Yu Gothic`, `Meiryo`, `sans-serif`
- 数字 / 英字: `IBM Plex Sans`, `Helvetica Neue`, `Arial`, `sans-serif`

主要サイズ:

| Role | Size | Weight | 用途 |
| --- | --- | --- | --- |
| Hero Title | `clamp(48px, 6vw, 84px)` | 800 | 半期評価レポート |
| Hero Company | `clamp(28px, 3.6vw, 58px)` | 800 | 会社名 |
| Hero Meta | `18px` | 700 | 期間、発行日、発行元 |
| Hero Lead | `20px` | 800 | ヒーロー要約 |
| Section Title | `22px` | 800 | セクション見出し |
| Panel Title | `19px` | 800 | パネル見出し |
| Body | `17px` | 400-700 | 本文、所見 |
| Table Text | `14px` | 700 | 表組み |
| KPI Number | `58-72px` | 700 | 指標カードの主数値 |

文字ルール:

- 数字は `font-variant-numeric: tabular-nums`
- 日本語は字間を広げすぎない
- 強調は色よりもサイズ、ウェイト、余白で作る

### 4-3. 角丸・線・影

| Element | Rule |
| --- | --- |
| ページ外枠 | `border-radius: 20px` |
| パネル / KPI / カード | `border-radius: 16px` |
| 小バッジ | `border-radius: 8px` または `999px` |
| 主区切り線 | `1px solid var(--line-200)` |
| パネル線 | `1px solid var(--line-300)` |
| ページ影 | 大きめの柔らかいネイビー影 |
| パネル影 | 弱い浮き感を与える補助影 |

影の考え方:

- 立体感ではなく「紙面の層」を見せるために使う
- 強いグローや硬い黒影は使わない

## 5. レイアウトルール

### 5-1. ページ全体

- 最大幅: `min(1380px, calc(100vw - 28px))`
- 背景: 薄いブルーグレーの縦グラデーション
- 本体: 白い紙面を中央配置

### 5-2. ヒーロー

ヒーローは以下の役割を持つ。

- ブランド表示
- レポート名
- 対象会社名
- 発行情報
- 一文要約
- 金融レポートらしいチャートモチーフ
- 主要KPIの導入

実装ルール:

- 左側にコピー、右側に相場グラフィック
- 背景は `Navy` グラデーション + 薄いグリッド + ドット
- KPIカードはデスクトップでヒーロー下端にまたがって配置する

### 5-3. 本文エリア

- `report-body` は白背景
- 最初のセクションまで十分な上余白を確保する
- 各セクションは細い罫線で区切る
- 左右2カラムが基本だが、情報量に応じて単カラムへ落とす

### 5-4. 余白

- 大セクション上下: `18px` から `24px`
- パネル内余白: `16px` から `20px`
- セクション見出し下: `12px`
- カード間ギャップ: `14px` から `16px`

## 6. コンポーネント設計

### 6-1. `report-page`

ページ本体の外殻。

- 白背景
- 大きめ角丸
- 全体シャドウ
- 画面中央寄せ

### 6-2. `report-hero`

ページ冒頭の核。

含む要素:

- `.hero-brand`
- `.sample-badge`
- `.hero-copy`
- `.hero-art`
- `.hero-kpis`

ルール:

- 視認性最優先
- 文字量が増えてもタイトルと会社名の優先順位は崩さない
- 相場グラフィックは装飾であり、文字可読性を邪魔しない

### 6-3. `hero-kpis` / `metric-card`

主要指標カード群。

役割:

- ページを開いた直後に「何位で、どの程度強いか」を把握させる

ルール:

- 1カード1メッセージ
- アイコン、ラベル、数値の順に視線が流れる
- 数値は中央寄せで強く見せる
- 補足注記は1行程度に留める

### 6-4. `section-heading`

青い番号バッジ + セクション見出し。

ルール:

- すべての大セクションで統一
- 章番号は装飾ではなく情報ナビゲーション
- 色は `Signal Blue 700`

### 6-5. `panel`

白系の基本パネル。

用途:

- テーブル
- 比較チャート
- 所見ボックス
- 診断インサイト

ルール:

- 背景は完全な真っ白ではなく、ごく薄いグラデーション可
- 枠線は淡く、影も弱く

### 6-6. テーブル

対象:

- 総合評価サマリー
- 更新事項

ルール:

- ヘッダー行は薄い青グラデーション
- 本文は白地
- 数値列は中央揃え + 数字フォント
- コメント列は読みやすさ優先で左揃え

### 6-7. 比較チャート

対象:

- 競争ポジション
- カテゴリ診断

ルール:

- 対象会社は `Blue`
- 比較相手は `Teal`
- 軸やグリッドは主張しすぎない薄いグレー
- 棒の長さと差分数値の両方で比較可能にする

### 6-8. `point-card`

主要評価ポイントの4カード。

ルール:

- 各カードは1つの論点に絞る
- 見出しは短く、本文で補う
- 番号とアイコンで視線誘導する

### 6-9. `editorial-card`

編集部所見の3カード。

種類:

- `green`: 強み
- `amber`: 今後反映しやすい観点
- `blue`: 総括

ルール:

- 色面ではなく淡い色味で区別する
- 強い警告色にはしない
- 最後の総括カードだけ本文型でも成立するようにする

### 6-10. `report-footer`

留意事項とデータ出典の締め。

ルール:

- 「注意」だが恐くしない
- 出典は別ボックスで静かに整理する
- ページ末尾に必要な実務感を残す

## 7. レスポンシブ方針

今回のページで重要なのは、KPIカードの扱いである。

### 7-1. デスクトップ `> 1260px`

- `hero-kpis` はヒーロー下端にまたがる絶対配置
- 5列1行
- `report-body` 上部に余白を確保して重なりを受ける

### 7-2. 中間幅 `<= 1260px`

- `hero-kpis` は通常フローに戻す
- 3列
- ヒーロー本文を隠さないことを優先する

### 7-3. タブレット `<= 1040px`

- ヒーロー右側グラフィックを通常フローへ戻す
- `summary-layout`、`two-column-layout`、`diagnosis-layout`、`report-footer` は1カラム化

### 7-4. 小型タブレット `<= 820px`

- `hero-kpis` は2列
- `point-grid` は1列
- 本文左右余白は `16px`

### 7-5. モバイル `<= 560px`

- `hero-kpis` は1列
- `hero-meta` と `hero-lead` は `15px`
- ページ背景はよりシンプルにする

再発防止ルール:

- KPIカードが複数行に折り返す幅では、絶対配置を維持しない
- `hero-lead` がカードに隠れるレイアウトは不可

---

## 8. 文体・コピー方針

- 中立
- 分析的
- 丁寧
- 必要以上に営業色を出さない

見出しルール:

- 事実ベースで短く言い切る
- 効果や評価軸が読み取れる言葉を使う

本文ルール:

- 強みと改善余地を分けて書く
- 「絶対評価」ではなく「相対評価」であることを保つ
- 順位上昇を約束するような書き方はしない

---

## 9. 実装ポリシー

### 維持したいこと

- 1ファイル完結の扱いやすさ
- 外部依存なしで開けること
- アイコンはインラインSVGで完結すること
- テキスト差し替えがしやすい生HTMLであること

### 変更してよいこと

- 会社名
- 数値
- 所見テキスト
- 更新事項
- 一部の比較バー長

### 変更時に壊してはいけないこと

- ヒーローの情報優先順位
- `Blue / Teal` の比較関係
- セクション番号のナビゲーション性
- テーブルと本文の読み分け
- モバイルでの `hero-lead` 可読性

## 10. 実装との対応

主要クラスの意味を以下に固定する。

| Class | Role |
| --- | --- |
| `.report-page` | 紙面全体 |
| `.report-hero` | 冒頭ヒーロー |
| `.hero-copy` | タイトル、会社名、メタ、要約 |
| `.hero-art` | 右側チャート装飾 |
| `.hero-kpis` | KPIカード群 |
| `.metric-card` | 個別KPIカード |
| `.report-body` | 本文領域 |
| `.section-block` | 各章ブロック |
| `.panel` | 汎用白パネル |
| `.competition-*` | 競争ポジション関連 |
| `.diagnosis-*` | カテゴリ診断関連 |
| `.point-card` | 評価ポイントカード |
| `.editorial-card` | 編集部所見カード |
| `.report-footer` | 注記 + データ出典 |

新規実装時の原則:

- 既存クラスを優先して再利用する
- 同じ役割に別名クラスを増やしすぎない
- デザイン差分はトークンとレイアウトで吸収する

---

## 11. このページの完成条件

以下を満たしていれば、このデザインは成立している。

- ヒーローだけで要点が把握できる
- KPIカードが本文を隠さない
- 表、比較バー、本文の3種類が視覚的に読み分けられる
- 総括と留意事項まで読むと、社内共有資料としてそのまま使える
- デスクトップでもモバイルでも、紙面感と信頼感が崩れない

以上を、このレポートHTMLのデザイン基準とする。
